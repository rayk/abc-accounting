# Dart Contract-First Workspace Pattern

## Overview

This pattern produces a three-package Dart workspace for contract-first development of a bounded domain capability. The core idea is that the behavioral contract — the interface, its vocabulary types, and its executable conformance suite — can be authored, proven self-consistent, and handed to an AI agent (ALCA) before a single line of implementation exists.

**The workspace is a development structure, not a release structure.** At publish time a collapse script merges `{name}` and `{name}_impl` into a single released package. Consumers see one package, one import, no registration call.

The three workspace packages are:

- `{name}` — the interface package: vocabulary types, abstract class, workspace-level registration factory
- `{name}_impl` — the implementation: concrete class, functional core, effects — **workspace-only, never published independently**
- `{name}_contract` — the conformance kit: harness input, ALCA steering — **workspace-only, never published**

The single released package is also named `{name}`. It contains the merged source of `{name}` and `{name}_impl`, a static factory (no `register()` call required), and none of the conformance infrastructure.

**When to use this pattern.** Apply it to any bounded domain capability that:
- Needs a structural compile-time guarantee that conformance tests cannot see the implementation
- Is developed with an AI agent (ALCA) in the implementation loop
- Is published as a single consumed artifact with no visible implementation details

## Package Structure

### Workspace (development)

```
{workspace-root}/
  pubspec.yaml                  ← workspace root (not a Dart package)
  pubspec.lock                  ← single shared lock for all three packages (committed)
  harness.yaml                  ← harness configuration (domain, package names, impl class)
  .harness_scratch/             ← generated by harness; gitignored; never committed
  tool/
    collapse.dart               ← release script: merges {name} + {name}_impl → single package
  {name}/
    pubspec.yaml                ← publish_to: omitted (managed by collapse script)
    lib/
      {name}.dart               ← public barrel (interface + vocabulary + workspace factory)
      src/
        contract/
          {name}.dart           ← token-guarded abstract class + UnimplementedXxx
          vocabulary.dart       ← value objects, errors, events, IDs, state
          version.dart          ← contractVersion constant
        registration/
          factory.dart          ← workspace-only: dynamic factory + register() + StateError guard
  {name}_impl/
    pubspec.yaml                ← publish_to: none  (workspace-only, never published)
    lib/
      {name}_impl.dart          ← workspace barrel: register() for harness wiring
      src/
        api/
          {name_impl_class}.dart  ← concrete class that extends {name_abstract_class}
        core/                   ← pure functional logic
        effects/                ← I/O seams (repository, clock, etc.)
          env.dart              ← XxxEnv typedef + defaultEnv()
          repository.dart       ← abstract XxxRepository interface
        runtime/                ← wiring (controller, coordinator)
        di/                     ← optional Riverpod providers
        registration.dart       ← contractImplemented constant + register()
    test/
      …unit tests…
      architecture_test.dart    ← dart_arch_test layer rules
      # impl_conformance_test.dart is generated by the harness — never committed
  {name}_contract/
    pubspec.yaml                ← publish_to: none  (workspace-only, never published)
    dart_test.yaml
    lib/
      {name}_contract.dart      ← public barrel
      src/
        types/
          type_config.dart      ← ContractBrief typeOverrides + formatNotes per vocabulary type
        brief/
          {domain}_brief.dart   ← ContractBrief instance: signatureRef, withContext(), domainType()
        contracts/              ← single-signature contracts (numbered, immutable, additive)
          001_open.dart         ← initial state contract  (tag: 'open')
          002_deposit.dart      ← deposit signature       (tag: 'deposit')
          003_withdraw.dart     ← withdraw signature      (tag: 'withdraw')
          …one file per interface method…
        scenarios/              ← multi-operation stateful sequences (numbered, immutable, additive)
          001_{theme}.dart      ← first cross-signature scenario (tag: 'scenario', 'scenario_{theme}')
          …one file per scenario…
        properties/             ← contract-level property tests (glados); excluded from ALCA inner loop
          arbitraries.dart      ← Arbitrary<T> for domain types; used only within {name}_contract
          001_{operation}_props.dart  ← property tests for one operation (tag: 'property', '{operation}')
          …one file per operation with properties…
        conformance.dart        ← thin aggregator: calls all contract + scenario functions
        reference_{name}.dart   ← ReferenceXxx: in-memory reference implementation
        {domain}_checks.dart    ← package:checks extension matchers
        switch.dart             ← XxxUnderTest settable factory switch
    test/
      pending_conformance_test.dart
      reference_conformance_test.dart
      honest_red_test.dart
      contract_version_test.dart
      brief_integrity_test.dart
```

**Workspace dependency direction (strict):**

```
{name}_impl      →  {name}          (workspace path dep)
{name}_contract  →  {name}          (workspace path dep)
harness          →  both            (runtime only, generated binding, scratch location)
```

`{name}_contract` has zero knowledge of `{name}_impl`. This is the defining structural constraint of the pattern.

### Release (collapsed single package)

The collapse script (`tool/collapse.dart`) runs at publish time and produces:

```
.release/{name}/
  pubspec.yaml          ← merged deps from {name} + {name}_impl; no {name}_impl dep
  lib/
    {name}.dart         ← merged public barrel
    src/
      contract/         ← from {name}/lib/src/contract/ (unchanged)
      api/              ← from {name}_impl/lib/src/api/
      core/             ← from {name}_impl/lib/src/core/
      effects/          ← from {name}_impl/lib/src/effects/
      runtime/          ← from {name}_impl/lib/src/runtime/
      di/               ← from {name}_impl/lib/src/di/ (if present)
      factory.dart      ← REPLACES registration/factory.dart: static wire-up, no register()
  test/
    …value-type unit tests from {name}/test/ only…
```

The dynamic registration factory is replaced by a static one:

```dart
// .release/{name}/lib/src/factory.dart  (generated by collapse script)
import 'api/{name_impl_class}.dart';
import 'effects/env.dart';
import 'contract/{name}.dart';

/// Returns a factory pre-wired to [{name_impl_class}].
/// No [register()] call required.
XxxFactory get defaultFactory => (id) => XxxImpl.open(_defaultEnv(), id);

XxxEnv _defaultEnv() => (
  repo: InMemoryXxxRepository(),
  clock: const SystemClock(),
  ids: MonotonicIdGenerator(const SystemClock()),
);
```

**Consumer view (release):**

```
consumers  →  {name}   (one package, one import, no register() call)
```

## Package 1: `{name}` — The Interface Package

### What it contains

- The token-guarded abstract base class (the domain interface)
- All vocabulary types: value objects, error hierarchy, event hierarchy, ID types, state, status
- `XxxFactory` typedef (the DI seam)
- `XxxResult` typedef (the return type of all commands)
- `contractVersion` constant
- `UnimplementedXxx` — the minimal concrete stub (zero overrides)
- A **workspace-only** dynamic registration factory (`registration/factory.dart`) — used by the harness during development; replaced by a static factory in the collapsed release artifact

### What it does NOT contain

- Any concrete implementation class
- `riverpod`, `fpdart`, or any framework dependency beyond what appears in the public API (see pubspec section)
- `NonEmptyChain` or any validation accumulator — those are implementation concerns that belong in `{name}_impl`
- `fast_immutable_collections` — if a vocabulary type needs a collection, use `dart:core` lists or a type with no heavy transitive deps
- Any test infrastructure other than `UnimplementedXxx`

> **Workspace vs release distinction.** The dynamic `register()` / `registerFactory()` mechanism in `registration/factory.dart` exists only to let the harness wire `{name}_impl` during development. In the collapsed release artifact this file is replaced by a static `factory.dart` that returns a pre-wired `XxxImpl` factory directly. The `register()` function does not appear in the released package — consumers never call it.

> **Breaking change from the reference exemplar.** The reference `abc_accounting` package defines `NonEmptyChain<E>` in `errors.dart` and imports `IList` from `fast_immutable_collections`, making that library a transitive dependency of every consumer of the interface. Moving `NonEmptyChain` and its dependency to `{name}_impl` is a deliberate breaking change from the exemplar structure. When migrating an existing bounded package to this pattern, treat the removal of `NonEmptyChain` from the vocabulary as a MAJOR version bump.

### What it does NOT contain (sealed types note)

Every error subtype in `vocabulary.dart` is final. The sealed class is defined here. Adding a new sealed subtype later is a **MAJOR** version bump — it breaks every consumer that exhaustively switches over the sealed type. Plan the sealed error hierarchy carefully before the initial release; additions after 1.0.0 incur a major version bump.

### Rationale for `UnimplementedXxx` living here

Downstream test authors writing integration tests against the `{name}` interface — in packages that are neither `{name}_impl` nor `{name}_contract` — need a zero-work test double. Requiring them to take a dev dependency on `{name}_contract` (a dev-only kit never published to pub.dev) just to get a 3-line stub class is excessive. `UnimplementedXxx` carries no logic and no dependencies. It belongs in `{name}`.

> **Tradeoff acknowledgment.** Placing test infrastructure (`UnimplementedXxx`) in a production package creates friction for teams with strict conventions that disallow test-only exports in non-dev packages (e.g., those using bnd_lints rules). The cleaner DDD alternative is a separately published `{name}_testing` package, analogous to `riverpod_testing`. For most teams the pragmatic tradeoff is worth it; teams with stricter conventions should publish a separate `{name}_testing` package and remove `UnimplementedXxx` from `{name}`.

### The token-guarded base class

```dart
// lib/src/contract/{name}.dart
import 'dart:async';
import 'package:fpdart/fpdart.dart';
import 'vocabulary.dart';

/// The outcome of any command: async, either the new [XxxState] or a [XxxError].
typedef XxxResult = Future<Either<XxxError, XxxState>>;

/// The seam for obtaining an [Xxx]. Conformance specs bind against this; they
/// never name a concrete type.
typedef XxxFactory = Future<Xxx> Function(XxxId id);

/// The domain operation contract — a token-guarded base class.
///
/// Implementations must `extend Xxx` and pass [Xxx.token] to the constructor.
/// A class that merely `implements Xxx` cannot obtain the token and fails
/// [verify]. Every member has a default body that throws [UnimplementedError],
/// so a method added here later does not break existing implementations — they
/// inherit the new default until they override it.
abstract class Xxx {
  Xxx({required Object token}) {
    _tokens[this] = token;
  }

  static final Expando<Object> _tokens = Expando<Object>('Xxx.token');
  static final Object _token = Object();

  /// The shared token a concrete [Xxx] must pass to its super constructor.
  static Object get token => _token;

  /// Throws unless [instance] was constructed by extending [Xxx] with the
  /// shared [token]. Called by the conformance harness for every SUT.
  ///
  /// NOTE: This is framework infrastructure, not domain logic. Application code
  /// never calls [verify]. Only the conformance harness calls it.
  static void verify(Xxx instance) {
    if (!identical(_tokens[instance], _token)) {
      throw AssertionError(
        'An Xxx must `extend Xxx` and pass `Xxx.token`; '
        '${instance.runtimeType} did not.',
      );
    }
  }

  // --- domain operations; each throws UnimplementedError by default ---

  XxxId get id => throw UnimplementedError();
  XxxState get state => throw UnimplementedError();
  Stream<XxxState> get changes => throw UnimplementedError();

  XxxResult doSomething(XxxValue value,
      {Option<CommandId> idempotencyKey = const None()}) =>
    throw UnimplementedError();

  Future<void> dispose() => throw UnimplementedError();
}

/// The intrinsic stub: extends [Xxx] with [Xxx.token] and overrides nothing.
/// Every member inherits the [UnimplementedError] default.
///
/// Use this in tests that need a minimal double of the [Xxx] interface without
/// depending on [package:{name}_contract].
final class UnimplementedXxx extends Xxx {
  UnimplementedXxx() : super(token: Xxx.token);
}
```

**Token-guard note.** The `_tokens` Expando, `_token` Object, and `verify()` method are framework boilerplate, not domain design. They enforce the federated-plugin contract (must extend, not just implement) and are called only by the conformance harness. Application code never calls `Xxx.verify()`. This block can be copied verbatim for every bounded package — the only change is the class name.

### The `contractVersion` constant

```dart
// lib/src/contract/version.dart
const String contractVersion = '0.1.0';
```

This one-line file is the runtime source of truth for the contract version. The collapse script reads it directly to validate alignment with `contractImplemented` and to populate the generated `factory.dart` header comment. Always keep it in sync with `{name}/pubspec.yaml`.

### The registration factory

```dart
// lib/src/registration/factory.dart
import '../contract/{name}.dart';

XxxFactory? _factory;

/// Returns the currently registered factory, or throws [StateError].
///
/// A [StateError] here means the harness did not call [registerFactory] before
/// running the conformance suite. This symbol does not exist in the released
/// package — the released [defaultFactory] is statically wired and never throws.
XxxFactory get defaultFactory {
  final f = _factory;
  if (f == null) {
    throw StateError(
      'No {name} implementation registered in this isolate. '
      'The harness must call `{name}_impl.register()` in setUpAll before '
      'running the conformance suite.',
    );
  }
  return f;
}

/// Whether any factory has been registered in this isolate (workspace-only).
/// Does not exist in the released package.
bool get isFactoryRegistered => _factory != null;

/// Register [factory] as the [Xxx] provider for this isolate.
void registerFactory(XxxFactory factory) => _factory = factory;
```

Export both `factory.dart` and the contract types from the top-level barrel:

```dart
// lib/{name}.dart
export 'src/contract/{name}.dart';
export 'src/contract/vocabulary.dart';
export 'src/contract/version.dart';
export 'src/registration/factory.dart';
```

### pubspec.yaml

```yaml
name: {name}
description: >-
  The {domain} interface: the token-guarded {Xxx} contract, vocabulary types,
  and contractVersion. No implementation; consumable directly from pub.dev or
  a git tag.
version: 0.1.0
# publish_to: omitted — this IS released; set when ready to publish

environment:
  sdk: ">=3.5.0 <4.0.0"

resolution: workspace   # required for Dart 3.5 workspace membership

dependencies:
  fpdart: ^1.2.0        # for Either/Option in the public API surface only
  # NO riverpod, NO fast_immutable_collections, NO framework deps

  test: ^1.25.0
  very_good_analysis: ^10.2.0
```

**Dependency constraint.** `{name}` must have the smallest possible dependency footprint. Every dep here becomes a transitive dep for every consumer. If a vocabulary type uses `Option` or `Either`, `fpdart` is appropriate. Otherwise, aim for zero non-hosted dependencies. `riverpod` never appears here. `fast_immutable_collections` never appears here (its use for validation accumulation belongs in `{name}_impl`).

## Package 2: `{name}_impl` — The Implementation Package

> **Workspace-only. `publish_to: none`. Never published independently.**
> Its source is merged into the released `{name}` package by the collapse script at publish time.

### What it contains

- The concrete implementation class (extends `Xxx` with `Xxx.token`)
- A `register()` function — **workspace/harness wiring only**; does not appear in the released package
- A `contractImplemented` constant — used by the harness version guard; collapsed into the static factory at release
- Effect seams: `XxxEnv` typedef, repository interface + in-memory implementation, clock, ID generator
- The pure functional core: decide, evolve, validation, commands
- Validation accumulators like `NonEmptyChain` and their dependencies (`fast_immutable_collections`)
- Optional: Riverpod providers

### What it does NOT contain

- The abstract base class or any vocabulary types — those live in `{name}`
- Any `bnd_eac` dependency — that lives in `{name}_contract`
- Any consumer-facing API — the public barrel only exports `register()` for the harness

### The workspace barrel

```dart
// lib/{name}_impl.dart
/// Workspace-only barrel. Not part of the released package.
///
/// Exports [register] so the harness-generated binding can wire [defaultFactory]
/// before running the conformance suite. This barrel and the [register] function
/// do not exist in the collapsed release artifact.
library;

export 'src/registration.dart' show register;
```

### The registration file (workspace-only)

```dart
// lib/src/registration.dart
import 'package:{name}/{name}.dart';
import 'api/{name_impl_class}.dart';
import 'effects/env.dart';

/// Which contract MAJOR.MINOR this implementation fully satisfies.
/// The harness version guard checks this against [contractVersion] from [{name}].
/// The collapse script reads this value to generate the static [defaultFactory].
const String contractImplemented = '0.1.0';

String _majorMinor(String v) => v.split('.').take(2).join('.');

/// Wires [defaultFactory] in [{name}] to [{name_impl_class}] for this isolate.
///
/// Called only by the harness-generated binding. Never called by application
/// code — in the released package [defaultFactory] is statically wired and
/// [register] does not exist.
void register() {
  if (_majorMinor(contractVersion) != _majorMinor(contractImplemented)) {
    throw StateError(
      'Workspace version mismatch: {name}_impl contractImplemented '
      '$contractImplemented vs {name} contractVersion $contractVersion. '
      'Their MAJOR.MINOR segments must agree.',
    );
  }
  registerFactory((id) => XxxImpl.open(defaultEnv(), id));
}

/// The default environment for workspace development and testing.
/// The collapse script promotes this to [_defaultEnv] in the released package.
XxxEnv defaultEnv() => (
  repo: InMemoryXxxRepository(),
  clock: const SystemClock(),
  ids: MonotonicIdGenerator(const SystemClock()),
);
```

### The `XxxEnv` record typedef

```dart
// lib/src/effects/env.dart
import 'package:{name}/{name}.dart';
import 'repository.dart';

/// The dependency record threaded through [{name_impl_class}.open].
///
/// All I/O effects are accessed through named fields of this record — no
/// ambient globals, no service locator. The concrete types here live in
/// {name}_impl/lib/src/effects/; the abstract interfaces (XxxRepository) are
/// defined alongside them.
typedef XxxEnv = ({
  XxxRepository repo,
  Clock clock,
  IdGenerator ids,
});
```

`XxxRepository` is the abstract interface defined at `{name}_impl/lib/src/effects/repository.dart`. `InMemoryXxxRepository` (its in-memory implementation), `SystemClock`, and `MonotonicIdGenerator` are defined in the same `effects/` directory. Their exact signatures are domain-specific; the pattern requires that all I/O seams pass through the `XxxEnv` record so they are fully substitutable in tests.

### The concrete class

```dart
// lib/src/api/{name_impl_class}.dart
import 'package:{name}/{name}.dart';
import '../effects/env.dart';
import '../runtime/controller.dart';

/// The concrete [{Xxx}] — must `extend Xxx` and pass `Xxx.token`.
/// Relative imports here remain valid after the collapse merge.
final class XxxImpl extends Xxx {
  XxxImpl._(this._controller) : super(token: Xxx.token);

  final XxxController _controller;

  static Future<Xxx> open(XxxEnv env, XxxId id) async =>
    XxxImpl._(await XxxController.create(env, id));

  @override
  XxxId get id => _controller.id;

  @override
  XxxState get state => _controller.state;

  @override
  Stream<XxxState> get changes => _controller.changes;

  @override
  XxxResult doSomething(XxxValue value,
      {Option<CommandId> idempotencyKey = const None()}) =>
    _controller.doSomething(value, idempotencyKey: idempotencyKey);

  @override
  Future<void> dispose() => _controller.dispose();
}
```

**`XxxController`** is the internal runtime coordination class, defined at `{name}_impl/lib/src/runtime/controller.dart`. It holds the current domain state and dispatches commands through the pure `core/` layer. Its structure is domain-specific: typically it holds mutable state (a reference to the current `XxxState`) and exposes one method per domain command, delegating pure decisions to `core/`. `XxxImpl.open()` delegates construction entirely to `XxxController.create(env, id)`. `XxxImpl` method overrides are thin wrappers that forward to `_controller`.

### impl_conformance_test.dart

**This file is a harness-generated artifact — it is not authored by the ALCA and is not committed to the repository.**

In the harness-mediated model the implementation ALCA has no knowledge of `{name}_contract`. The harness owns the boundary crossing: it generates a temporary binding, runs the contract tests, extracts `=== BRIEF ===` output, and streams it to the ALCA. The ALCA only edits `{name}_impl/lib/src/api/{name_impl_class}.dart` in response to what the harness streams.

The harness generates the following binding into the scratch directory each run:

```dart
// GENERATED BY HARNESS — do not commit, do not edit
@TestOn('vm')
library;

import 'package:{name}_contract/{name}_contract.dart';
import 'package:{name}_impl/{name}_impl.dart' as impl;
import 'package:{name}/{name}.dart';
import 'package:test/test.dart';

void main() {
  setUpAll(() {
    impl.register();
    xxxBrief.forImplementation(
      file: '{name}_impl/lib/src/api/{name_impl_class}.dart',
    );
  });

  xxxAcceptance('{name_impl_class}', defaultFactory);

  test('{name_impl_class} passes the token verification', () async {
    final sut = await defaultFactory(const XxxId('v'));
    expect(() => Xxx.verify(sut), returnsNormally);
    await sut.dispose();
  });
}
```

**Why the harness owns this.** The binding is entirely mechanical — it follows from the workspace naming convention. `{name}_impl` exports `register()`, which wires `defaultFactory`. `{name}_contract` exports `xxxAcceptance()` and `xxxBrief`. The `File:` path is derivable from the workspace structure. There is no decision for a human or ALCA to make here; the harness generates it from convention.

**What the harness needs to derive the binding:**

| Value | Source |
|-------|--------|
| `xxxAcceptance` function name | Convention: `{domain}Acceptance` |
| `xxxBrief` name | Convention: `{domain}Brief` |
| `impl.register()` | Exported by `{name}_impl` barrel |
| `defaultFactory` | Exported by `{name}` registration module |
| `File:` path | Workspace structure + naming convention |
| `XxxId` | Exported by `{name}` barrel |

The SUT label passed to `xxxAcceptance` is the concrete class name (e.g. `'LedgerImpl'`), not the package name. This label appears in test output — the class name is more informative in failure reports.

The harness configuration in the workspace root (see **Harness** section) provides the `{domain}` value and any non-conventional paths.

## Package 3: `{name}_contract` — The Conformance Kit

### What it contains

| Area | Path | Contents |
|------|------|----------|
| **Type config** | `src/types/type_config.dart` | `typeOverrides` and `formatNotes` maps for every vocabulary type used in ContractBrief |
| **Brief** | `src/brief/{domain}_brief.dart` | `ContractBrief` instance: `signatureRef`, `withContext()`, `domainType()` registrations — imports `type_config.dart` |
| **Contracts** | `src/contracts/NNN_{operation}.dart` | One file per interface method — numbered, immutable, additive. Each tagged at file/group/test level |
| **Scenarios** | `src/scenarios/NNN_{theme}.dart` | One file per stateful sequence — numbered, immutable, additive. Cross-signature, state-carrying |
| **Aggregator** | `src/conformance.dart` | Thin: imports all contracts + scenarios, calls each in order inside `xxxAcceptance()` |
| **Properties** | `src/properties/arbitraries.dart` | `Arbitrary<T>` for domain types — used only within `{name}_contract` (not imported by `{name}_impl`) |
| **Properties** | `src/properties/NNN_{op}_props.dart` | Contract-level property tests via `glados`; tagged `property`; excluded from ALCA inner loop |
| **Matchers** | `src/{domain}_checks.dart` | `package:checks` extension methods for vocabulary types |
| **Reference** | `src/reference_{name}.dart` | `ReferenceXxx`: minimal in-memory implementation that proves the spec is self-consistent |
| **Switch** | `src/switch.dart` | `XxxUnderTest`: settable factory for harness wiring |

Note: `UnimplementedXxx` is exported from `{name}`, not from `{name}_contract`. The contract kit re-exports it via its barrel for ergonomic access, but the canonical home is `{name}`.

### The additive-migration principle

Contracts and scenarios are **never modified after authoring**. They accumulate like database migration scripts:
- Adding a new interface method → create `NNN_{method}.dart` in `contracts/`, add one line to `conformance.dart`
- Adding a new stateful sequence → create `NNN_{theme}.dart` in `scenarios/`, add one line to `conformance.dart`
- Discovered missing rejection case → new contract file with the next number; do NOT edit the existing file

The number prefix (`001_`, `002_`) encodes authoring order and makes the file list self-documenting as a history of the spec's growth.

### Why `{name}_contract` depends ONLY on `{name}`

This single constraint provides the compile-time boundary guarantee. Because `{name}_contract` has no dependency on `{name}_impl`, any attempt to import an implementation type in conformance kit code is a hard compile error. The conformance suite is structurally guaranteed to be implementation-blind.

In the reference exemplar this guarantee is convention-only: `contracts_for_abc_accounting` path-depends on `abc_accounting` which contains the implementation, relying on the developer to import only `abc_accounting_contract.dart`. The new structure makes the compiler the enforcer.

### The `lib/` vs `test/` split

`xxxAcceptance()` lives in `lib/` (not `test/`) because the harness imports it cross-package when generating and running the binding. In Dart, test files in `test/` are not importable by other packages. Any artifact that the harness must import must be in `lib/`.

### The public barrel

```dart
// lib/{name}_contract.dart
library;

// Contract-only re-export: conformance kit authors against the {name} interface;
// implementation types stay out of scope.
export 'package:{name}/{name}.dart';

export 'src/conformance.dart';          // xxxAcceptance() aggregator
export 'src/{domain}_checks.dart';      // package:checks matchers
export 'src/brief/{domain}_brief.dart'; // xxxBrief — required by the harness-generated binding
export 'src/reference_{name}.dart';
export 'src/switch.dart';

// Individual contract and scenario files are NOT exported from the barrel.
// They are internal to conformance.dart. The harness targets them by tag, not by import.
```

**xxxBrief must be in the barrel.** The harness imports only the main barrel and calls `xxxBrief.forImplementation()`. If `src/brief/{domain}_brief.dart` is not exported, the harness-generated binding fails to compile.

**Individual contract/scenario files are not barrel-exported.** They are imported only by `conformance.dart`. External callers (harness, test files) use `xxxAcceptance()` for the full suite or tag filtering for focused runs — they never import contract files directly.

**4. `test/contract_version_test.dart`** — the release gate
```dart
@TestOn('vm')
library;

import 'dart:io';

import 'package:{name}/{name}.dart' show contractVersion;
import 'package:test/test.dart';

/// Read the version: field from a pubspec.yaml, anchored to this test file's
/// location rather than CWD. Throws [StateError] if the file is missing
/// or contains no version: field.
String _versionOf(Uri pubspecUri) {
  final file = File.fromUri(pubspecUri);
  if (!file.existsSync()) {
    throw StateError('cannot find pubspec at ${file.path}');
  }
  final match = RegExp(r'^version:\s*(\S+)', multiLine: true)
      .firstMatch(file.readAsStringSync());
  if (match == null) throw StateError('no `version:` in ${file.path}');
  return match.group(1)!;
}

void main() {
  // Anchor all paths to this test file, not to CWD.
  // __FILE__ is at: {workspace}/{name}_contract/test/contract_version_test.dart
  // {name} pubspec is at: {workspace}/{name}/pubspec.yaml
  // {name}_contract pubspec is at: {workspace}/{name}_contract/pubspec.yaml
  final testDir = Uri.file(Platform.script.toFilePath()).resolve('../');
  final contractPubspecUri = testDir.resolve('pubspec.yaml');
  final interfacePubspecUri = testDir.resolve('../{name}/pubspec.yaml');

  final contractVersion_ = _versionOf(contractPubspecUri);
  final interfaceVersion = _versionOf(interfacePubspecUri);

  // Extract MAJOR.MINOR only — PATCH releases of {name}_contract are allowed
  // to carry new conformance scenarios or infrastructure improvements without
  // requiring a {name} release.
  String majorMinor(String v) => v.split('.').take(2).join('.');

  test('{name} and {name}_contract share the same MAJOR.MINOR', () {
    expect(
      majorMinor(interfaceVersion),
      majorMinor(contractVersion_),
      reason:
          '{name} ($interfaceVersion) and {name}_contract ($contractVersion_) '
          'must share MAJOR.MINOR. Bump both when the contract changes.',
    );
  });

  test('the runtime contractVersion constant agrees with the {name} package version', () {
    // contractVersion lives in {name}/lib/src/contract/version.dart.
    // It must match the {name} pubspec version — not the {name}_contract version,
    // which may diverge at PATCH level. Test 1 above already guards the
    // cross-package MAJOR.MINOR relationship.
    expect(
      contractVersion,
      interfaceVersion,
      reason:
          'The runtime contractVersion ($contractVersion) must equal the '
          '{name} pubspec version ($interfaceVersion). '
          'Update {name}/lib/src/contract/version.dart.',
    );
  });
}
```

**Why test #2 compares against `interfaceVersion`, not `contractVersion_`.** The `contractVersion` constant lives in `{name}`, so it should match `{name}`'s version. `{name}_contract` may release a PATCH (e.g., 0.1.0 → 0.1.1 to add a new conformance scenario or fix infrastructure) without requiring a `{name}` release. After that PATCH: `interfaceVersion = '0.1.0'`, `contractVersion_ = '0.1.1'`. Test 1 passes (same MAJOR.MINOR: `0.1 == 0.1`). Test 2 passes (`contractVersion '0.1.0' == interfaceVersion '0.1.0'`). Under the old comparison (`contractVersion == contractVersion_`), test 2 would fail — making `{name}_contract` PATCH independence structurally impossible.

**Why `_versionOf` uses `throw StateError`, not `fail()`.** `fail()` from `package:test` must be called inside a `test()` body. Calling it outside a test body (e.g., in `main()` at the top level) propagates `TestFailure` as an unhandled exception, causing the test runner to report a load error rather than a named failing test. `throw StateError(...)` produces a clear diagnostic regardless of call site.

**Critical fix: Platform.script anchor.** The reference exemplar uses `File('pubspec.yaml')` and `File('../pubspec.yaml')` — CWD-relative paths. In the 3-package workspace, `../pubspec.yaml` from `{name}_contract/test/` resolves to the workspace root (which has no `version:` field), silently reading the wrong file. The `Platform.script.toFilePath()` anchor fixes this regardless of what directory `dart test` is invoked from.

**6. `impl_conformance_test.dart`** — this file does NOT exist in `{name}_contract/test/` or `{name}_impl/test/` as a committed file. It is generated at runtime by the harness. See Package 2 and the **Harness** section.

### The Rejection Coverage Principle

> For each interface method, enumerate every `XxxError` subtype reachable from that method. Each typed rejection case must be its own conformance group — with `setRule()` naming the error type and `filterTypes()` including it. A method with only a happy-path group is an ALCA steering defect: the ALCA implements the method, all tests pass, and the rejection path is never written because there is no failing test to steer it there.

**Minimum groups per contract file:**
- One happy-path group (success, state transition verified)
- One group per distinct `XxxError` subtype the method can return
- One idempotency group if the method accepts a key

### Naming conventions

**Contract and scenario files** — numbered for additive ordering, snake\_case operation name:

```
contracts/001_open.dart
contracts/002_deposit.dart
contracts/003_deposit_idempotency.dart   ← separate file if idempotency added later
scenarios/001_account_lifecycle.dart
scenarios/002_daily_limit_enforcement.dart
```

**Group names** — `'{operation} — {clause}'` in title case:

```dart

group('deposit — AmountMustBePositive', ...)
group('deposit — idempotency', ...)
group('account lifecycle — freeze blocks money movement', ...)
```

**Test names** — complete sentence describing the observable outcome:

```dart
test('balance increases by the deposited amount', ...)
test('zero amount returns AmountMustBePositive', ...)
test('negative amount returns AmountMustBePositive and leaves state unchanged', ...)
```

**Tags** — three-level hierarchy:

| Level | Pattern | Example |
|---|---|---|
| Contract file | `{operation}` | `'deposit'` |
| Group within contract | `'{operation}_{clause}'` | `'deposit_amount_positive'` |
| Individual test | `'{operation}_{clause}_{variant}'` | `'deposit_amount_positive_zero'` |
| Scenario file | `'scenario'` | `'scenario'` |

### `src/brief/{domain}_brief.dart` — ContractBrief

```dart
// lib/src/brief/{domain}_brief.dart
// NOTE: @TestOn('vm') does NOT belong here. lib/ file annotations have no effect
// on the test runner. Add @TestOn('vm') to each test/ file that calls xxxAcceptance().
library;

import 'package:{name}/{name}.dart';
import 'package:bnd_eac/brief.dart';
import 'package:fpdart/fpdart.dart';
import '../types/type_config.dart';

/// Representative signature for mirror extraction — reflected, never called.
/// Gives dart:mirrors a concrete named function to inspect for the Function: line.
XxxResult xxxOperation(
  XxxValue amount, {
  Option<CommandId> idempotencyKey = const None(),
}) =>
    throw UnimplementedError('boundary scaffold — reflected, never run');

/// The steering brief for the {Xxx} contract.
/// Installed in the outer setUp() of xxxAcceptance(). Silent when green;
/// renders === BRIEF === on any failure. The harness calls
/// xxxBrief.forImplementation(file: ...) to redirect File: to the impl class.
final xxxBrief = ContractBrief(
  signatureRef: xxxOperation,
  function: xxxOperation,
)
  ..withContext(
    domain: '{Domain}',
    package: '{name}_impl',
    interfacePath: '{name}/lib/src/contract/{name}.dart',
    version: contractVersion,
  )
  ..domainType<XxxState>(
    typeOverrides: xxxStateOverrides,
    formatNotes: xxxStateFormatNotes,
  )
  ..domainType<XxxInsufficientFunds>(
    typeOverrides: insufficientFundsOverrides,
  )
  ..domainType<AmountMustBePositive>(
    typeOverrides: amountMustBePositiveOverrides,
  );
  // Add ..domainType<T>() for every sealed error subtype in vocabulary.dart.
```

**ContractBrief constructor parameters.**

- `signatureRef`: reflected at brief installation time via `dart:mirrors` to generate the `Function:` and `TYPEDEF:` lines. Never called at runtime. Must be a named top-level function — mirrors require a concrete function object.
- `function`: same function value in standard use. Pass the same to both parameters unless the mirror target and the runtime callable must differ (rare edge case).

**`withContext()`.** Configures the CONTEXT preamble on the first failure of a run. Omitting it means a cold-start ALCA receives no orientation. The four fields map directly to the CONTEXT preamble fields in **ALCA Steering**.

**`typeOverrides` key validation.** Keys are not compile-time validated. `brief_integrity_test.dart` provides the runtime guard — one test per `domainType` with `typeOverrides`. When renaming a vocabulary field, update `type_config.dart` and rerun `brief_integrity_test.dart`.

### `src/contracts/NNN_{operation}.dart` — individual contract files

Each file is self-contained: its own SUT lifecycle, its own brief configuration per group, all groups for that signature. The file is **never modified after authoring** — changes and additions are new files.

```dart
// lib/src/contracts/002_deposit.dart
@Tags(['deposit'])           // file-level tag — applies to all groups and tests
library;

import 'package:{name}/{name}.dart';
import 'package:checks/checks.dart';
import 'package:test/test.dart';
import '../brief/{domain}_brief.dart';
import '../{domain}_checks.dart';

/// Contract for [Xxx.deposit].
/// Authored: {date}. Never modified after initial authoring.
void depositContract(XxxFactory factory) {
  late Xxx sut;
  setUp(() async => sut = await factory(const XxxId('deposit')));
  tearDown(() async => sut.dispose());

  group('deposit — happy path', () {
    setUpAll(() => xxxBrief
      ..setRule('A positive deposit increases the balance by the deposited amount.')
      ..filterTypes({XxxState, XxxValue}));

    test('balance increases by deposited amount', () async {
      check(await sut.deposit(const XxxValue(500)))
          .success.balance.equals(const XxxValue(500));
    }, tags: 'deposit_happy_balance');

    test('version advances after deposit', () async {
      check(await sut.deposit(const XxxValue(100)))
          .success.version.equals(const Version(1));
    }, tags: 'deposit_happy_version');

  }, tags: 'deposit_happy');

  group('deposit — AmountMustBePositive', () {
    setUpAll(() => xxxBrief
      ..setRule(
        'Zero or negative amount is rejected with AmountMustBePositive. '
        'State is unchanged.',
      )
      ..filterTypes({AmountMustBePositive, XxxValue, XxxState}));

    test('zero amount returns AmountMustBePositive', () async {
      final before = sut.state;
      check(await sut.deposit(const XxxValue(0)))
          .failure.isA<AmountMustBePositive>();
      check(sut.state).equals(before);
    }, tags: 'deposit_amount_positive_zero');

    test('negative amount returns AmountMustBePositive', () async {
      check(await sut.deposit(const XxxValue(-1)))
          .failure.isA<AmountMustBePositive>();
    }, tags: 'deposit_amount_positive_negative');

  }, tags: 'deposit_amount_positive');

```

**SUT lifecycle per contract.** Each contract function owns its own `setUp`/`tearDown` — a fresh `Xxx` is created before every `test()`. `setUpAll` inside each group configures `xxxBrief` once per group. This keeps rejection tests valid: "state is unchanged" assertions start from a pristine SUT with no prior mutations.

**Circular import trap.** Contract files import `brief/{domain}_brief.dart` via relative path, never via the package barrel. The barrel exports `conformance.dart`; `conformance.dart` imports contract files; contract files must not import the barrel — that would create a cycle. Always use relative imports inside `lib/src/`.

### `src/scenarios/NNN_{theme}.dart` — stateful scenario files

Scenarios differ from contracts in one key way: **SUT state carries forward between groups within the scenario**. The setup deposits funds that the next group then tries to withdraw; the freeze in group 2 is the state the close group in group 3 operates on.

```dart
// lib/src/scenarios/001_account_lifecycle.dart
@Tags(['scenario'])                    // all scenario files carry this
library;

import 'package:{name}/{name}.dart';
import 'package:checks/checks.dart';
import 'package:test/test.dart';
import '../brief/{domain}_brief.dart';
import '../{domain}_checks.dart';

/// Stateful lifecycle scenario: open → deposit → freeze → close.
/// Exercises state machine transitions across multiple signatures in sequence.
/// Authored: {date}. Never modified after initial authoring.
void accountLifecycleScenario(XxxFactory factory) {
  late Xxx sut;
  // One SUT for the whole scenario — state carries forward across groups.
  setUpAll(() async => sut = await factory(const XxxId('lifecycle')));
  tearDownAll(() async => sut.dispose());

  group('account lifecycle — freeze blocks money movement', () {
    setUpAll(() => xxxBrief
      ..setRule(
        'After freeze(), deposit() is rejected with AccountNotActive. '
        'freeze() is idempotent.',
      )
      ..filterTypes({AccountNotActive, XxxState}));

    test('deposit establishes pre-freeze balance', () async {
      await sut.deposit(const XxxValue(200));
      check(sut.state).balance.equals(const XxxValue(200));
    }, tags: 'scenario_lifecycle_prefund');

    test('freeze() is idempotent', () async {
      await sut.freeze();
      await sut.freeze(); // second call must not throw
    }, tags: 'scenario_lifecycle_freeze_idempotent');

    test('deposit after freeze returns AccountNotActive', () async {
      check(await sut.deposit(const XxxValue(10)))
          .failure.isA<AccountNotActive>();
      check(sut.state).balance.equals(const XxxValue(200)); // unchanged
    }, tags: 'scenario_lifecycle_freeze_blocks_deposit');

  }, tags: 'scenario_lifecycle');

  group('account lifecycle — close is terminal', () {
    setUpAll(() => xxxBrief
      ..setRule(
        'After closeAccount(), status is closed and money movement '
        'is permanently rejected with AccountNotActive.',
      )
      ..filterTypes({AccountNotActive, XxxState}));

    test('closeAccount() sets status to closed', () async {
      await sut.closeAccount();
      check(sut.state).status.equals(AccountStatus.closed);
    }, tags: 'scenario_lifecycle_close_status');

    test('deposit after close returns AccountNotActive', () async {
      check(await sut.deposit(const XxxValue(10)))
          .failure.isA<AccountNotActive>();
    }, tags: 'scenario_lifecycle_close_blocks_deposit');

  }, tags: 'scenario_lifecycle');
}
```

**`setUpAll`/`tearDownAll` vs `setUp`/`tearDown`.** Scenarios use `setUpAll`/`tearDownAll` at the outer level — one SUT shared across all groups. This is intentional: the scenario tests state machine transitions, so prior group state IS the precondition for the next group. Contracts use `setUp`/`tearDown` — fresh SUT per test — because isolation is the invariant there.

### `src/properties/` — contract-level property tests

Property tests specify invariants that must hold for arbitrarily generated inputs, not just the examples in contracts and scenarios. They live in `{name}_contract` because the invariants ARE part of the spec.       ..setRule('Any non-positive amount is always rejected regardless of account state.')
      ..filterTypes({AmountMustBePositive, XxxValue}));

}

### The `XxxChecks` extension matchers

`{domain}_checks.dart` provides `package:checks` extension methods for `Either<XxxError, XxxState>` (the awaited result of any domain command) and for `XxxState` (domain-specific field accessors). These extensions are required by the conformance suite — without them, the `.success`, `.failure`, and domain-field accessors used in group test bodies do not compile.

```dart
// lib/src/{domain}_checks.dart
import 'package:checks/checks.dart';
import 'package:{name}/{name}.dart';
import 'package:fpdart/fpdart.dart';

/// Fluent assertions for the awaited result of a domain command:
///   Either<XxxError, XxxState>
///
/// Usage in conformance tests:
///   check(await sut.deposit(XxxValue(500))).success.balance.equals(XxxValue(500));
///   check(await sut.withdraw(XxxValue(9999))).failure.isA<InsufficientFunds>();
extension XxxChecks on Subject<Either<XxxError, XxxState>> {
  /// Asserts the result is [Right] (success) and returns a [Subject<XxxState>]
  /// for further assertions on the returned state.
  Subject<XxxState> get success => has(
    (result) => result.fold(
      (err) => throw AssertionError(
          'Expected Right (success) but got Left (failure): $err'),
      (state) => state,
    ),
    'is a success (Right)',
  );

  /// Asserts the result is [Left] (failure) and returns a [Subject<XxxError>]
  /// for further assertions on the error type and fields.
  Subject<XxxError> get failure => has(
    (result) => result.fold(
      (err) => err,
      (state) => throw AssertionError(
          'Expected Left (failure) but got Right (success): $state'),
    ),
    'is a failure (Left)',
  );
}

/// Domain-specific field accessors for [XxxState].
///
/// Add one [has()] accessor per [XxxState] field that conformance tests
/// assert on directly. The accessor name mirrors the field name.
extension XxxStateChecks on Subject<XxxState> {
  Subject<XxxValue> get balance => has((s) => s.balance, 'balance');
  // Add further accessors as the domain model grows.
}
```

**Authoring guidance.** `XxxChecks` extends `Subject<Either<XxxError, XxxState>>` — the type that `check(await sut.someCommand(...))` produces. `XxxStateChecks` extends `Subject<XxxState>` — the type that `.success` returns. Each `has()` call takes an extractor function and a label; if the extractor throws, `package:checks` reports the label as the failed expectation description. Domain-specific error subtypes (e.g., `InsufficientFunds`) are checked using the standard `.isA<T>()` extension already provided by `package:checks` — no custom extension is needed for subtype checks.

### The `XxxUnderTest` settable factory switch

`switch.dart` provides the settable factory switch used in `reference_conformance_test.dart` to exercise the stub and reference paths independently.

```dart
// lib/src/switch.dart
import 'package:{name}/{name}.dart';
import 'reference_{name}.dart';

XxxFactory? _factory;

/// The settable factory switch for contract-level tests.
///
/// Tests that exercise the reference path call [useReference()] and must call
/// [useStub()] in their [tearDown] to restore the default. The pattern in
/// [reference_conformance_test.dart]:
///
///   group('...', () {
///     tearDown(XxxUnderTest.useStub);
///     test('...', () async {
///       XxxUnderTest.useReference();
///       ...
///     });
///   });
class XxxUnderTest {
  XxxUnderTest._();

  /// Returns the current factory.
  /// Defaults to [UnimplementedXxx] if [useReference] has not been called.
  static XxxFactory get factory =>
      _factory ?? (_) async => UnimplementedXxx();

  /// Resets the factory to the unimplemented stub (the default state).
  /// Call in [tearDown] after any test that calls [useReference].
  static void useStub() => _factory = null;

  /// Switches the factory to [ReferenceXxx.open].
  /// Call at the start of a test body before exercising the SUT.
  static void useReference() => _factory = ReferenceXxx.open;
}
```

## Workspace Root

### `pubspec.yaml`

```yaml
# Workspace root — NOT a Dart package. No `name:` field.
environment:
  sdk: ">=3.5.0 <4.0.0"

workspace:
  - {name}
  - {name}_impl
  - {name}_contract
```

**`resolution: workspace` is required in every member.** Each member package's `pubspec.yaml` must include `resolution: workspace`. Without this field, `dart pub get` on an individual package resolves independently with its own lock rather than joining the shared workspace lock. The workspace root's `workspace:` key alone is not sufficient — the opt-in must be in each member.

**Always run `dart pub get` from the workspace root.** Running `dart pub get` inside a member directory (e.g., inside `{name}_impl/`) creates a member-level `pubspec.lock` scoped to that package, potentially shadowing the shared workspace lock and causing resolution divergence that is difficult to diagnose. Always run `dart pub get` at `{workspace-root}/`. Gitignore per-package lock files in member directories.

**The shared lock file.** The workspace resolver produces a single `pubspec.lock` at the workspace root. Commit this file. It guarantees that `{name}`, `{name}_impl`, and `{name}_contract` resolve the same transitive versions for shared dependencies (e.g., `fpdart`, `checks`, `test`). The coupling the shared lock creates (a dep bump in `{name}_impl` can affect `{name}`'s transitive resolution) is intentional: this is the workspace's value.

### How local resolution works during development

During development all three packages are resolved through the workspace lock. The `path:` dependency in `{name}_contract` pointing to `../{name}` is resolved through the workspace resolver, not independently. No `dependency_overrides` are needed. The `{name}` path dependency in `{name}_impl` is resolved to the local workspace member directly — `{name}` does not need to be published to pub.dev before working on `{name}_impl` in the workspace.

### Release mechanism

The workspace is a development-only structure. At release time, run `tool/collapse.dart` — it produces the single publishable `{name}` artifact at `.release/{name}/`. `{name}_impl` and `{name}_contract` are never published independently. See the **Collapse** and **Versioning and Release** sections for the full publish procedure.

## The Contract-First Lifecycle

### Phase 0: Bootstrap

1. Create the workspace root directory.
2. Create `workspace_root/pubspec.yaml` with the `workspace:` key (no `name:`).
3. Create the three package directories with minimal `pubspec.yaml` files, each containing `resolution: workspace`.
4. Run `dart pub get` **at the workspace root** to create `pubspec.lock`. Never run it inside a member directory.

### Phase 1: Contracting (no implementation exists)

**Owner: contract author (human or contracting ALCA)**

1. Define the interface in `{name}/lib/src/contract/{name}.dart`:
   - Abstract base with token guard
   - All domain operations with `UnimplementedError` default bodies
   - `UnimplementedXxx` stub (3 lines, no overrides)
2. Define all vocabulary types in `{name}/lib/src/contract/vocabulary.dart`:
   - Only types named by the interface's method signatures and return types
   - No validation accumulators; no `NonEmptyChain`; no `fast_immutable_collections`
   - Plan the sealed error hierarchy carefully — adding a new sealed subtype after 1.0.0 is a MAJOR bump
3. Set `contractVersion = '0.1.0'` in `{name}/lib/src/contract/version.dart`.
4. Author the conformance suite in `{name}_contract/lib/src/conformance.dart`:
   - For each command: enumerate all reachable `XxxError` subtypes
   - One group per happy path AND one group per typed rejection case — apply the Rejection Coverage Principle
   - `setRule()` text must name all preconditions and invariants, including error type names
5. Author `ReferenceXxx` in `{name}_contract/lib/src/reference_{name}.dart`:
   - Naive in-memory implementation; no optimization, no persistence
   - Hold state in a `final Map<XxxId, XxxState>` (or equivalent) initialised to empty
   - Return `Future.value(...)` for all async operations — no real async needed
   - Validate preconditions with direct `if`/`else` returning `Either.left(XxxError(...))` — no validation accumulator
   - Implements only what conformance groups directly exercise
   - Must be readable by a domain novice in under 10 minutes
6. Verify contracting completeness:
   ```bash
   dart test test/reference_conformance_test.dart \
              test/honest_red_test.dart \
              test/contract_version_test.dart
   ```
   All three must be **green**. This is the phase-complete gate.
7. Optionally verify the red state:
   ```bash
   dart test --run-skipped -t pending
   ```
   All scenarios must be **red** (each throwing `UnimplementedError`).
8. Tag the completed contract: `git tag {name}/v0.1.0` (see tagging strategy).
9. Publish `{name}` to pub.dev or push the git tag.

**The phase-complete gate is machine-verifiable.** A CI required check on steps 6 ensures the implementation ALCA receives a self-consistent, executable spec. This is not documentation — it is the handover artifact.

### Phase 2: Implementation

**Owner: implementation ALCA (or human implementer)**
**Harness: runs continuously, streams failing briefs to the ALCA**

The ALCA has no knowledge of `{name}_contract`. It receives a stream of `=== BRIEF ===` blocks from the harness and implements the named clause. The harness owns test execution entirely.

**ALCA responsibilities:**
1. Read the interface barrel (`{name}/lib/src/contract/{name}.dart`) to learn the full type vocabulary before writing any code.
2. Author the concrete class skeleton in `{name}_impl/lib/src/api/{name_impl_class}.dart`: `extends Xxx` with `Xxx.token`, `register()` in `lib/src/registration.dart` with `contractImplemented = '0.1.0'`.
3. Verify `Xxx.verify(sut)` would pass by confirming `extends Xxx` with correct token — the harness's first run checks this and streams `AssertionError` if wrong.
4. Implement each method clause as the harness streams failing briefs, one group at a time, until the harness stream goes silent (all green).
5. After harness goes silent, run architecture tests:
   ```bash
   dart test test/architecture_test.dart
   ```

**Harness responsibilities (runs continuously, invisible to ALCA):**
1. Watches `{name}_impl/lib/` for file changes.
2. On change: generates the binding, runs the contract tests against the current impl.
3. Extracts the first `=== BRIEF ===` block from the failure output.
4. Streams the brief to the ALCA's context.
5. When all tests pass: streams a silent green signal — no brief, no output.

See the **Harness** section for full details on discovery, binding generation, and output filtering.

### Phase 3: Evolution (adding a new method)

**When contractVersion and contractImplemented diverge.** After bumping `contractVersion` in `{name}` (step 1 of any evolution cycle), `register()` in `{name}_impl` immediately throws `StateError` because the MAJOR.MINOR segments no longer agree. The harness cannot run conformance tests until alignment is restored. This is expected behavior — not a bug.

**Required sequence to restore harness operation after a version bump:**
1. Bump `contractVersion` in `{name}/lib/src/contract/version.dart` and `{name}/pubspec.yaml`.
2. Bump `{name}_contract` version (same MINOR as `{name}`).
3. **Update `contractImplemented` in `{name}_impl/lib/src/registration.dart` to match the new MAJOR.MINOR** — this is the step that unblocks the harness. Do this before attempting to implement the new method.
4. Restart the harness. With `contractImplemented` updated, `register()` succeeds and the harness begins streaming the first brief for the new method.
5. Implement as usual in response to briefs.

The ALCA must update `contractImplemented` as the first action in any evolution cycle. The harness stream is blocked (register() throws) until this update is made.

**Evolution checklist:**

1. Add the new method to the interface in `{name}` with a `throw UnimplementedError()` default body. Existing `{name}_impl` continues to compile — it inherits the new default.
2. Add new conformance groups in `{name}_contract` for the new method's behavior, applying the Rejection Coverage Principle.
3. Bump `{name}` and `{name}_contract` versions together (same MINOR bump). Update `contractVersion`.
4. **Update `contractImplemented` in `{name}_impl/lib/src/registration.dart` to the new MAJOR.MINOR** — this unblocks the harness (register() stops throwing StateError). Do this before implementing the new method or running any harness step.
5. Verify: `reference_conformance_test` must be green with `ReferenceXxx` implementing the new method. `pending_conformance_test` shows the new groups as red.
6. Run `dart run tool/collapse.dart` — alignment check passes only after `{name}_impl` has been updated (step 4 above) and the new method is implemented.
7. `dart pub publish --directory .release/{name}`
8. `git tag v{version} && git push origin v{version}`

> **Optional capabilities marker.** If a future method or behavior is declared optional (e.g., an implementation may declare it does not support idempotency keys), `xxxAcceptance()` will need a `capabilities` parameter to allow an implementation to opt out of those groups. Design the capabilities parameter at the same time as the optional feature is introduced — do not attempt to retrofit it. An implementation that legitimately omits an optional feature will have permanently failing conformance groups with no sanctioned workaround if the parameter is absent.

**Policy: a `{name}` MINOR release is always accompanied by a same-day collapse and publish.** The collapse script enforces alignment between `contractVersion` (in `{name}`) and `contractImplemented` (in `{name}_impl`) before producing the release artifact — if `{name}_impl` has not been updated to implement the new method, the collapse script fails loudly before anything is published. There is no window of consumer exposure to a partially-implemented release.

## ALCA Steering

### How the brief works

`ContractBrief` is installed in the outer `setUp()` of `xxxAcceptance()`. Each scenario group's `setUpAll` calls `setRule()` and `filterTypes()` to scope the brief to that group's behavioral clause.

When a test is green: the brief is completely silent. No output, no overhead.

When a test fails: the brief renders a `=== BRIEF ===` block in the test failure output. On the **first failure in a run**, a CONTEXT preamble is rendered to orient a cold-start ALCA that has not previously seen this bounded package:

```
═══ BRIEF ═══
CONTEXT (rendered on first failure only):
  Domain: {Domain}
  Package: {name}_impl
  Interface: package:{name}/{name}.dart — {name}/lib/src/contract/{name}.dart
  ContractVersion: 0.1.0

File: {name}_impl/lib/src/api/{name_impl_class}.dart
Function: xxxOperation(XxxValue amount, {Option<CommandId> idempotencyKey}) → XxxResult
TYPEDEF: XxxResult = Future<Either<XxxError, XxxState>>
CONTRACT: A withdrawal exceeding the balance is rejected with InsufficientFunds
          and leaves state unchanged.
TYPES:
  sealed XxxError
    ← XxxInsufficientFunds(required XxxValue balance, required XxxValue requested)
  XxxState(required XxxId id, required XxxValue balance, ...)
FIELD-FORMATS:
  XxxState.balance — integer minor units; XxxValue is a newtype over int
═══════════════
```

**CONTEXT preamble.** The preamble block — Domain, Package, Interface, ContractVersion — is rendered once on the first failure of a run. It provides the minimal orientation a cold-start ALCA needs: what domain this is, which package to edit, where the full interface surface lives, and which contract version is in play. Subsequent failures in the same run omit the preamble. This is configured via the `withContext()` call chain in the `xxxBrief` definition — see the ContractBrief pattern section above.

**TYPEDEF section.** The `TYPEDEF:` line expands the result typedef inline. Without this, an ALCA reading only the brief does not know that `XxxResult = Future<Either<XxxError, XxxState>>` and may construct the wrong return type. If `bnd_eac` does not yet render this automatically, explicitly direct the ALCA to read the interface barrel as step 0 of the implementation loop to learn the full type vocabulary.

**Sealed hierarchy in TYPES.** When a registered domainType is a subtype of a sealed class, the brief renders the sealed parent as a one-line header (`sealed XxxError`) above the subtype. This tells the ALCA how to construct `Either.left(XxxInsufficientFunds(...))` and that the sealed hierarchy must be exhaustively matched in switch expressions.

The `File:` line in contracting phase points to `{name}/lib/src/contract/{name}.dart`. The harness reconfigures it for the implementation phase by calling `xxxBrief.forImplementation(file: ...)` in the generated binding's `setUpAll` — the ALCA never touches the brief configuration.

### The ALCA implementation loop

The ALCA's loop is deliberately narrow. It does not know that `{name}_contract` exists.

1. ALCA reads the interface barrel (`{name}/lib/src/contract/{name}.dart`) — learns the full type vocabulary, typedef expansions, and method signatures before writing any code.
2. ALCA authors the skeleton: `XxxImpl extends Xxx` with `Xxx.token`, `register()` with `contractImplemented = '0.1.0'`.
3. Harness detects the file and begins streaming. First brief arrives — typically the token guard failure if `extends Xxx` is wrong, or the first scenario group if the skeleton is correct.
4. ALCA opens the `File:` path, implements the `CONTRACT:` clause using `TYPES:`, `TYPEDEF:`, and `FIELD-FORMATS:` as the type vocabulary.
5. ALCA saves the file. Harness re-runs. Next brief arrives (next failing group) or stream goes silent (all green).
6. ALCA repeats from step 4 until the harness stream is silent.
7. ALCA runs `dart test test/architecture_test.dart` — this is the only test the ALCA runs directly.

**ALCA prioritization.** The harness streams only the first failing brief per run, targeted at the failing group's tag. The ALCA implements that clause, the harness re-runs with the same tag, and only advances to the next tag when that group is green. Groups across contracts are authored in behavioral dependency order (`open` → `deposit` → `withdraw` → etc.), so stream order is always a valid implementation sequence. Scenarios run after all contracts are green.

**ALCA scope — behavioral vs. architectural correctness.** The `File:` line guarantees the ALCA opens the correct override class — it is the entry point, not a complete inventory of every file to change. For a multi-layer architecture (facade → controller → use-case → decider), implementing a method on the facade requires creating or modifying command types, decider branches, and use-cases. The ALCA discovers these from the directory structure and architecture test layer rules. Behavioral correctness (harness silent) and architectural correctness (arch test green) are sequential goals.

**ALCA scope — brief vs. architecture.** The brief names what to implement (signature, rule, types). It does not prescribe internals. Architecture violations are a second failure class with a different steering mechanism (`dart_arch_test`), not the brief.

### Discovery

The harness identifies the three workspace packages by role from the workspace `pubspec.yaml` member list and a harness configuration file at the workspace root:

```yaml

impl: {name}_impl          # package directory name

impl_class: {name_impl_class}  # e.g. "LedgerImpl" — the concrete class name
```

**`domain` casing rule.** The `domain` value must be PascalCase (e.g., `Ledger`, not `ledger`). The harness applies the following derivation rules:

| Derived name | Transformation | Example (`domain: Ledger`) |
|---|---|---|
| `xxxAcceptance` | lowercase first char | `ledgerAcceptance` |
| `xxxBrief` | lowercase first char | `ledgerBrief` |
| `XxxId` | unchanged (PascalCase) | `LedgerId` |

Storing `ledger` (lowercase) instead of `Ledger` as the domain value would derive `lEdgerId` from `{Domain}Id` if the harness naively uppercases — use PascalCase consistently to avoid transformation ambiguity.

Derived values (all follow from `domain`):
- `xxxAcceptance` → `{domain}Acceptance` (camelCase)
- `xxxBrief` → `{domain}Brief`
- `XxxId` → `{Domain}Id`

### Binding generation

On each run the harness writes a temporary binding to the scratch directory and compiles it with access to the workspace packages:

```dart
// GENERATED — .harness_scratch/test/impl_conformance_test.dart, never committed
@TestOn('vm')
library;

import 'package:{name}_contract/{name}_contract.dart';
import 'package:{name}_impl/{name}_impl.dart' as impl;
import 'package:{name}/{name}.dart';
import 'package:test/test.dart';

void main() {
  setUpAll(() {
    impl.register();
    {domain}Brief.forImplementation(
      file: '{name}_impl/lib/src/api/{name_impl_class}.dart',
    );
  });

  {domain}Acceptance('{name_impl_class}', defaultFactory);

  test('{name_impl_class} passes the token verification', () async {
    final sut = await defaultFactory(const {Domain}Id('v'));
    expect(() => {Domain}.verify(sut), returnsNormally);
    await sut.dispose();
  });
}
```

The generated file uses only exported symbols — nothing from `lib/src/`. The scratch directory is added to `.gitignore` and cleaned between runs.

### Scratch directory and test invocation

The harness writes the generated binding and its supporting pubspec to `{workspace-root}/.harness_scratch/`. This directory must be outside all three workspace packages so that its imports of `{name}`, `{name}_impl`, and `{name}_contract` do not create intra-workspace dependency cycles.

**Scratch directory pubspec** (generated by the harness on first run if absent):

```yaml
# .harness_scratch/pubspec.yaml  — generated; do not commit
name: _harness_scratch
publish_to: none
environment:
  sdk: ">=3.5.0 <4.0.0"
dependencies:
  {name}:
    path: ../{name}
  {name}_impl:
    path: ../{name}_impl
  {name}_contract:
    path: ../{name}_contract
  test: ^1.25.0
```

Path dependencies pointing to the workspace siblings allow `package:` imports in the generated binding to resolve without making `.harness_scratch/` a workspace member. The harness runs `dart pub get` inside `.harness_scratch/` after writing this file.

**dart test invocation** — three modes:

**Default inner loop** (every file-change event — fast, excludes property tests):
```bash
dart test test/impl_conformance_test.dart --reporter compact --exclude-tags property
```

**Focused run** (harness-selected after identifying the failing tag from prior output):
```bash
dart test test/impl_conformance_test.dart --reporter compact --exclude-tags property -t {failing_tag}
```

**Full suite including property tests** (CI — not triggered by the harness; run as a separate CI step):
```bash
dart test test/impl_conformance_test.dart --reporter compact
```

The harness always passes `--exclude-tags property` during the ALCA inner loop. Property tests are slow (many generated inputs) and do not contribute `=== BRIEF ===` blocks — they are not useful for the one-clause-at-a-time implementation loop.

The harness extracts the failing group's tag from the test name in the previous run's output, then re-runs with only that tag until the group is green before advancing to the next failing tag. This tightens the ALCA feedback loop to a single group.

The working directory for all commands is `.harness_scratch/`. The `--reporter compact` flag produces parseable per-test output.

**Add to workspace root `.gitignore`:**

```
.harness_scratch/
```

### Output filtering

Raw `dart test` output contains package loading, timing, and pass/fail lines. The harness parses the output and extracts only:

1. The first `═══ BRIEF ═══ ... ═══════════════` block
2. The `package:checks` failure diff immediately following the block

Everything else is discarded. The ALCA receives a compact context packet:

```
═══ BRIEF ═══
CONTEXT (first failure only):
  Domain: {Domain}
  Package: {name}_impl
  Interface: {name}/lib/src/contract/{name}.dart
  ContractVersion: 0.1.0

File: {name}_impl/lib/src/api/{name_impl_class}.dart
Function: xxxOperation(XxxValue amount, {Option<CommandId> idempotencyKey}) → XxxResult
TYPEDEF: XxxResult = Future<Either<XxxError, XxxState>>
CONTRACT: <the clause from setRule()>
TYPES:
  <filtered domain types from filterTypes()>
FIELD-FORMATS:
  <notes for in-scope types>
═══════════════

Expected: <checks failure diff>
Actual:   <checks failure diff>
```

When all tests pass the harness streams nothing — silence is the green signal.

### File watching

The harness watches `{name}_impl/lib/` recursively. On any `.dart` file change it debounces for 300ms (to avoid re-running mid-save) then runs the full binding. It does not watch `{name}_impl/test/` — the ALCA does not write test files.

### Isolation from the ALCA

The harness runs as a separate process. The ALCA interacts with it only through:
- Editing `{name}_impl/lib/` files (triggers a run)
- Receiving the streamed brief (the harness writes to the ALCA's context window)

The ALCA has no mechanism to invoke, configure, or inspect the harness. This boundary is intentional: it keeps the ALCA's context focused on implementation concerns and ensures the steering signal (the brief) is always harness-authored, never ALCA-authored.

### Harness and the structural boundary

The harness is the one component that imports both `{name}_contract` and `{name}_impl`. This is the legitimate boundary crossing point — it happens in a generated, never-committed file in a scratch directory, not in either workspace package. The three workspace packages remain structurally clean:

```
{name}_contract  →  {name}      (compile-time: no {name}_impl types)
{name}_impl      →  {name}      (compile-time: no {name}_contract types)
harness          →  both        (runtime only, generated binding, scratch location)
```

## Collapse

The collapse script (`tool/collapse.dart`) is the release mechanism. It merges `{name}` and `{name}_impl` into a single publishable package at `.release/{name}/`, validates the result, and leaves it ready for `dart pub publish`. It is the only step that produces a releasable artifact.

### What it does

```
1. Validate alignment      — contractVersion == contractImplemented (MAJOR.MINOR)
2. Create .release/{name}/ — clean output directory
3. Copy interface source   — {name}/lib/src/contract/ → .release/{name}/lib/src/contract/
4. Copy impl source        — {name}_impl/lib/src/{api,core,effects,runtime,di}/
                             → .release/{name}/lib/src/{api,core,effects,runtime,di}/
5. Generate static factory — replaces registration/factory.dart with a static wire-up
6. Merge pubspec.yaml      — union of deps; strips {name}_impl, bnd_eac, checks, dart_arch_test
7. Write merged barrel     — exports contract + vocabulary + static factory + (optional) Riverpod providers
8. Copy value-type tests   — {name}/test/ only; no conformance tests, no impl tests
                             (if {name}/test/ is absent, this step is a no-op — not an error)
9. dart analyze            — fails loud on any import error or type violation
10. dart test              — runs the copied value-type tests against the merged package
```

### Alignment validation

The script reads `contractVersion` from `{name}/lib/src/contract/version.dart` and `contractImplemented` from `{name}_impl/lib/src/registration.dart`. If their MAJOR.MINOR segments differ, the script exits with a clear error before touching the output directory:

```
ERROR: Cannot collapse — version mismatch.
  {name}        contractVersion    = 0.2.0  (MAJOR.MINOR: 0.2)
  {name}_impl   contractImplemented = 0.1.0  (MAJOR.MINOR: 0.1)
Update {name}_impl to implement contract 0.2 before collapsing.
```

The script extracts these constants by regex: `const String contractVersion = '(\S+)'` applied to the relevant source file. The constant format is intentionally constrained (single-quoted string, no interpolation) to make regex extraction reliable.

### Dependency conflict resolution

When merging `{name}/pubspec.yaml` and `{name}_impl/pubspec.yaml` dependencies, the script applies the following rule: if the same package appears in both with compatible version constraints, use the intersection (the narrower bound). For example, `fpdart: ^1.2.0` in `{name}` and `fpdart: ^1.3.0` in `{name}_impl` yields `fpdart: ^1.3.0` in the merged pubspec. If the same package appears with incompatible constraints (e.g., `^1.0.0` and `^2.0.0`), the script exits with an error listing the conflicting entries before any output files are written.

### Generated static factory

The script generates `factory.dart` from the values it reads during alignment validation:

```dart
// .release/{name}/lib/src/factory.dart  (generated — do not edit)
// Generated by tool/collapse.dart from:
//   {name} contractVersion    = {contractVersion}
//   {name}_impl contractImplemented = {contractImplemented}
import 'api/{name_impl_class}.dart';
import 'effects/env.dart';
import 'contract/{name}.dart';

/// Pre-wired factory for [{name_impl_class}]. No setup required.
XxxFactory get defaultFactory => (id) => XxxImpl.open(_defaultEnv(), id);

XxxEnv _defaultEnv() => (
  repo: InMemoryXxxRepository(),
  clock: const SystemClock(),
  ids: MonotonicIdGenerator(const SystemClock()),
);
```

**Import path note.** The file lives at `.release/{name}/lib/src/factory.dart`. Imports are relative to `lib/src/`, so `'api/{name_impl_class}.dart'` resolves to `lib/src/api/{name_impl_class}.dart` — no leading `src/` prefix. Using `'src/api/...'` would produce the path `lib/src/src/api/...` which does not exist and fails `dart analyze`.

**`_defaultEnv()` is private.** The `defaultEnv()` helper in the workspace `registration.dart` is public because it is useful to workspace development tooling. In the generated release factory, `_defaultEnv()` is private — it is not part of the consumer API and should not be callable from outside `factory.dart`.

### Merged pubspec.yaml

The script produces a pubspec that is the union of both packages' `dependencies`, with workspace-only and dev-only entries removed:

```yaml
# .release/{name}/pubspec.yaml  (generated — do not edit)
name: {name}
description: >-
  {description from {name}/pubspec.yaml}
version: {version from {name}/pubspec.yaml}

environment:
  sdk: ">=3.5.0 <4.0.0"

# Merged dependencies — workspace-only packages removed:
#   removed: {name}_impl (merged by source)
#   removed: bnd_eac, checks, dart_arch_test (dev-only tools)
dependencies:
  fpdart: ^1.2.0
  fast_immutable_collections: ^11.0.0  # from {name}_impl if present
  riverpod: ^2.6.0                     # from {name}_impl if present
```

### Merged barrel

```dart
// .release/{name}/lib/{name}.dart  (generated — do not edit)
export 'src/contract/{name}.dart';     // abstract class, XxxFactory, XxxResult, UnimplementedXxx
export 'src/contract/vocabulary.dart'; // value objects, sealed errors, state, IDs
export 'src/contract/version.dart';    // contractVersion
export 'src/factory.dart';             // defaultFactory (statically wired)
```

`src/contract/vocabulary.dart` must be exported explicitly. The abstract class file (`src/contract/{name}.dart`) only `import`s vocabulary.dart — it does not re-export it. Without this explicit export line, all sealed error subtypes, state types, value objects, and ID types are invisible to consumers of the released package.

An optional Riverpod barrel is generated if `{name}_impl/lib/src/di/` is non-empty:

```dart
// .release/{name}/lib/{name}_riverpod.dart  (generated if Riverpod providers exist)
export 'src/di/providers.dart';
```

### Running the collapse script

```bash
# From workspace root:
dart run tool/collapse.dart

# Inspect the result:
ls .release/{name}/lib/src/

# Dry-run publish check:
dart pub publish --directory .release/{name} --dry-run

# Publish:
dart pub publish --directory .release/{name}

# Tag:
git tag v{version} && git push origin v{version}
```

### What is excluded from the release

| Source | Reason excluded |
|---|---|
| `{name}_impl/` as a package | Merged by source; not a dependency |
| `{name}_contract/` | Dev-only conformance kit |
| `registration/factory.dart` | Replaced by generated static factory |
| `{name}_impl/lib/src/registration.dart` | Workspace-only alignment marker; `contractImplemented` and `register()` are excluded from the public API |
| `contractImplemented` constant | Workspace-internal; not in public API |
| `register()` / `registerFactory()` | Workspace harness wiring; not in public API |
| `isFactoryRegistered` | Workspace harness diagnostic; not in public API |
| `bnd_eac`, `checks`, `dart_arch_test` | Dev-only tools |
| Any `test/` from `{name}_impl` | Unit tests for internal impl; not shipped |
| `harness.yaml` | Workspace configuration |

## Versioning and Release

### The two-way version sync

`{name}` and `{name}_contract` jointly define the contract — the interface and its executable specification are inseparable. They version together on MAJOR.MINOR.

`contract_version_test.dart` asserts:

```
{name} pubspec MAJOR.MINOR == {name}_contract pubspec MAJOR.MINOR
contractVersion (runtime constant) == {name} pubspec version (full equality)
```

Full patch equality between `{name}` and `{name}_contract` is not required. `{name}_contract` may release PATCH versions (e.g., 0.1.0 → 0.1.1) for infrastructure improvements or new conformance scenarios without requiring a `{name}` release. `contractVersion` lives in `{name}` and matches the `{name}` pubspec version — it does not change on a `{name}_contract` PATCH release.

Adding a new conformance scenario that specifies previously-undocumented behavior IS a contract change — that is a MINOR bump to both `{name}` and `{name}_contract`.

`{name}_impl` versions independently. Its version never participates in the sync assertion.

### SemVer rules per package

**`{name}` MAJOR bump:**
- Any removal or incompatible change to the interface (remove a method, change a method signature, remove or rename a vocabulary type)
- **Adding a new subtype to a sealed vocabulary class** (e.g., a new `XxxError` variant). Every consumer that exhaustively switches over the sealed type gets a compile error on the next build. This is a breaking change regardless of whether anything else changes. Adding a non-sealed `final class` to the vocabulary (no sealed base) is MINOR, not MAJOR.

**`{name}` MINOR bump:** Addition to the interface (new non-sealed vocabulary type, new method). Existing workspace implementations keep compiling — they inherit `UnimplementedError` default. The workspace version guard in `register()` flags the mismatch until `{name}_impl` is updated. The collapse script enforces alignment before publishing.

**`{name}` PATCH bump:** Documentation, no behavioral change. The collapse script can run and publish without updating `{name}_impl`.

**`{name}_impl` does not have a published version.** It is workspace-only. Its `contractImplemented` constant is an internal alignment marker checked by the harness and the collapse script, not a semver published to consumers.

### Release ordering and enforcement

There is only **one** publish step: run `tool/collapse.dart`, then `dart pub publish` from `.release/{name}/`. The collapse script enforces alignment between `contractVersion` (in `{name}`) and `contractImplemented` (in `{name}_impl`) before producing the release artifact, failing loudly if they differ.

```bash
# Full release in three commands:
dart run tool/collapse.dart          # validates + produces .release/{name}/
dart pub publish --directory .release/{name} --dry-run   # final check
dart pub publish --directory .release/{name}
```

There is no second publish for `{name}_impl` and no pub.dev propagation delay to wait for.

### Git tagging strategy

One tag per release. The workspace produces one published package — use standard version tags:

```
v0.1.0       ← published {name} release (collapsed artifact)
v0.2.0       ← next release
```

For consumers using git dependencies:

```yaml
# Consumer pubspec.yaml
dependencies:
  {name}:
    git:
      url: https://github.com/yourorg/{repo}.git
      ref: "v0.1.0"
      path: .release/{name}   # or publish to pub.dev and use hosted dep
```

`{name}_impl` and `{name}_contract` never get release tags.

## Factory Pattern Details

### Consumer wiring (released package)

The released `{name}` package has a statically wired factory. Consumers need one package and zero setup calls:

```yaml
# Consumer's pubspec.yaml — one package only
dependencies:
  {name}: ^0.1.0
```

```dart
// Consumer's main.dart — no register() call, no {name}_impl import
import 'package:{name}/{name}.dart';

void main() {
  // defaultFactory is already wired — use it immediately
  runApp(MyApp());
}
```

```dart
// Application code — one import, full type safety
import 'package:{name}/{name}.dart';

Future<void> doWork(XxxId id) async {
  final xxx = await defaultFactory(id);
  final result = await xxx.doSomething(const XxxValue(100));
  result.fold(
    (err) => log(err.message),
    (state) => log('Success: ${state.balance}'),
  );
  await xxx.dispose();
}
```

### Workspace registration (harness wiring only)

The dynamic `register()` / `registerFactory()` mechanism exists only during workspace development to let the harness wire `{name}_impl.XxxImpl` into `{name}.defaultFactory` before running contract tests. It is a development-time seam, not a consumer-facing API.

The collapse script replaces `registration/factory.dart` in the release artifact with:

```dart
// Generated by tool/collapse.dart — static, no register() required
import 'api/{name_impl_class}.dart';
import 'effects/env.dart';
import 'contract/{name}.dart';

XxxFactory get defaultFactory => (id) => XxxImpl.open(_defaultEnv(), id);

XxxEnv _defaultEnv() => (
  repo: InMemoryXxxRepository(),
  clock: const SystemClock(),
  ids: MonotonicIdGenerator(const SystemClock()),
);
```

The `isFactoryRegistered`, `registerFactory()`, and `register()` symbols do not exist in the released package.

### Riverpod consumers (released package)

If the collapsed release includes Riverpod providers, a second barrel is available:

```dart
import 'package:{name}/{name}_riverpod.dart';

final container = ProviderContainer(
  overrides: [xxxEnvProvider.overrideWithValue(customEnv)],
);
final xxx = await container.read(xxxProvider(someId).future);
```

The collapse script includes `lib/src/di/providers.dart` from `{name}_impl` and generates the `{name}_riverpod.dart` barrel. If no Riverpod providers exist in `{name}_impl`, the barrel is not generated.

### When multiple instances co-exist

`defaultFactory` is a top-level getter — it returns a new factory call each time, not a singleton. Each call to `defaultFactory(id)` creates a fresh `XxxImpl` instance. For scoped multi-instance scenarios (e.g., different environment per tenant), use the Riverpod `ProviderContainer` with `xxxEnvProvider` overrides — each container is independent.

## Dependency Rules

| Package | Context | May depend on | May NOT depend on |
|---|---|---|---|
| `{name}` | workspace + release | `fpdart`, `dart:core`, `dart:async` | `{name}_impl`, `{name}_contract`, `riverpod`, `bnd_eac`, `fast_immutable_collections` |
| `{name}_impl` | workspace only | `{name}` (workspace path), `fpdart`, `fast_immutable_collections`, `riverpod` (optional), `glados` (dev), `dart_arch_test` (dev) | `{name}_contract` (any form — dev or production) |
| `{name}_contract` | workspace only | `{name}` (workspace path), `fpdart`, `checks`, `test`, `bnd_eac`, `glados` | `{name}_impl` (structural boundary — compiler enforced) |
| released `{name}` | release only | merged production deps of `{name}` + `{name}_impl` | `{name}_contract`, `bnd_eac`, `glados`, `test`, any dev-only tool |
| consumers | post-release | released `{name}` only | `{name}_impl` (not published), `{name}_contract` (not published) |

The `{name}_impl` / `{name}_contract` boundary is absolute — no direction, no form. The harness is the only legitimate crossing point, and it operates from `.harness_scratch/` outside both packages. `architecture_test.dart` enforces this as a CI-required check.

### Cross-bounded-context dependencies in `{name}_impl`

If `{name}_impl` depends on another bounded context (e.g., a reporting service or an auth service), the following rule applies:

Any dependency of `{name}_impl` on another bounded context package MUST be mediated by an abstract interface class defined inside `{name}_impl` (not in `{name}`). The concrete adapter implementing that interface is package-private to `{name}_impl`. The upstream bounded context's types never appear in `{name}`'s vocabulary and never in `{name}_impl`'s public barrel.

Example structure:

```
{name}_impl/lib/src/effects/
  repository.dart           ← abstract XxxRepository (defined here, not in {name})
  reporting_adapter.dart    ← concrete adapter calling the reporting package (private)
```

The architecture test in `{name}_impl` asserts that `lib/src/effects/` may import external bounded context packages, and `{name}/lib/` may not.

### Architecture test

```dart
// {name}_impl/test/architecture_test.dart
//
// Verifies layer rules for the {name}_impl package.
// Uses dart_arch_test ^0.3.1 — verify exact method names against that package's
// documentation if the API below does not match your installed version.
import 'package:dart_arch_test/dart_arch_test.dart';
import 'package:test/test.dart';

void main() {
  group('layer rules — {name}_impl', () {
    test('core/ is pure — no I/O, no framework, no coupling to api/ or effects/', () {
      // The core/ layer contains pure functional logic only.
      // It must not import anything from effects/, api/, or runtime/.
      expectLayerRule(
        sourceLayer: 'lib/src/core',
        mustNotImportLayers: [
          'lib/src/effects',
          'lib/src/api',
          'lib/src/runtime',
        ],
        package: '{name}_impl',
      );
    });

    test('api/ must not reach into effects/ directly', () {
      // The api/ layer (concrete class) delegates to runtime/ and core/.
      // It must not bypass the runtime layer by importing effects/ directly.
      expectLayerRule(
        sourceLayer: 'lib/src/api',
        mustNotImportLayers: ['lib/src/effects'],
        package: '{name}_impl',
      );
    });

    test('no lib/ source imports {name}_contract', () {
      // {name}_contract has zero dependency relationship with {name}_impl — not
      // even as a dev dep. This test guards against accidental imports in lib/.
      // If this fires, check that no file in lib/src/ has a package:{name}_contract import.
      expectNoProductionImport(
        importPrefix: 'package:{name}_contract/',
        package: '{name}_impl',
      );
    });
  });
}
```

**Note on dart_arch_test API.** The method names `expectLayerRule` and `expectNoProductionImport` above are illustrative — they follow the conceptual model of `dart_arch_test ^0.3.1` but you must verify exact method signatures against the installed package's documentation. The three rules to enforce are: (1) `core/` has no I/O or framework coupling; (2) `api/` does not import `effects/` directly; (3) no `lib/` file imports `{name}_contract`.

**When architecture_test.dart fails.** The failure output names the file containing the violating import and the rule that was broken (e.g., `lib/src/api/ledger_impl.dart imports lib/src/effects/repository.dart — violates rule api/ mustNotImport effects/`). To fix: locate the import in the named file, determine which layer the imported symbol belongs in logically, and either move the symbol to the correct layer or introduce an abstraction boundary between the layers. Re-run the test after fixing.

## Naming Conventions

### Package naming

| Role | Name pattern | Example |
|---|---|---|
| Interface | `{domain}` | `ledger`, `payments`, `inventory` |
| Implementation | `{domain}_impl` | `ledger_impl` |
| Conformance kit | `{domain}_contract` | `ledger_contract` |

Avoid `_platform_interface` suffix (Flutter plugin convention) — it implies a platform-specific concern. The `_contract` suffix reads as "executable contract" which is more accurate for this pattern.

### File naming

| Content | File | Location |
|---|---|---|
| Abstract class | `{domain}.dart` | `{name}/lib/src/contract/` |
| Vocabulary types | `vocabulary.dart` | `{name}/lib/src/contract/` |
| `contractVersion` | `version.dart` | `{name}/lib/src/contract/` |
| Registration factory | `factory.dart` | `{name}/lib/src/registration/` |
| Concrete class | `{name_impl_class}.dart` | `{name}_impl/lib/src/api/` |
| `register()` | `registration.dart` | `{name}_impl/lib/src/` |
| `XxxEnv` typedef | `env.dart` | `{name}_impl/lib/src/effects/` |
| Type mirror config | `type_config.dart` | `{name}_contract/lib/src/types/` |
| ContractBrief | `{domain}_brief.dart` | `{name}_contract/lib/src/brief/` |
| Single-sig contract | `NNN_{operation}.dart` | `{name}_contract/lib/src/contracts/` |
| Stateful scenario | `NNN_{theme}.dart` | `{name}_contract/lib/src/scenarios/` |
| Contract property tests | `NNN_{operation}_props.dart` | `{name}_contract/lib/src/properties/` |
| Contract arbitraries | `arbitraries.dart` | `{name}_contract/lib/src/properties/` |
| Impl-internal arbitraries | `arbitraries.dart` | `{name}_impl/test/` |
| Acceptance aggregator | `conformance.dart` | `{name}_contract/lib/src/` |
| Reference impl | `reference_{domain}.dart` | `{name}_contract/lib/src/` |
| SUT switch | `switch.dart` | `{name}_contract/lib/src/` |
| Checks matchers | `{domain}_checks.dart` | `{name}_contract/lib/src/` |
| typeOverrides test | `brief_integrity_test.dart` | `{name}_contract/test/` |
| Architecture test | `architecture_test.dart` | `{name}_impl/test/` |

**Contract/scenario numbering.** Use three-digit zero-padded numbers (`001_`, `002_`). Numbers encode authoring order and sort files chronologically in any directory listing. Gaps are allowed — if `003_` is reserved for future use or a file was superseded, skip to `004_`. Never renumber existing files.

### Type naming

| Role | Convention | Example |
|---|---|---|
| Abstract base | `{Domain}` (no suffix) | `Ledger`, `Payments` |

| Concrete impl | `{Domain}Impl` or `Account{Domain}` | `LedgerImpl`, `AccountLedger` |
| SUT switch | `{Domain}UnderTest` | `LedgerUnderTest` |
| Result typedef | `{Domain}Result` | `LedgerResult` |
| Factory typedef | `{Domain}Factory` | `LedgerFactory` |
| Error base | `{Domain}Error` | `LedgerError` |
| Event base | `{Domain}Event` | `LedgerEvent` |
| State type | `{Domain}State` | `AccountState` |
